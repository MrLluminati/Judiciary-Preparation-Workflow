# Research Report Integration Reference

## Source

Reference file supplied by learner: `deep-research-report on judiciary study workflow.md`.

## Purpose of This File

This file records how the research report should influence the Judiciary Preparation Workflow repository.

The report should be treated as a documentation blueprint and reference layer, not as a replacement for the daily coaching workflow.

## Core Direction from the Report

The repository should function as a **study operating system for a practicing advocate preparing for Bihar Judiciary**, not as a generic law-notes archive.

The system must remain built around:

1. court-compatible study routine;
2. three-block study day;
3. 12-week subject sequence;
4. subject priority map;
5. article/section-first study;
6. bare-act-first learning;
7. daily output through MCQs or answer writing;
8. revision ladder and error log.

## Documentation Principles to Follow

### 1. Docs-first

The repository must make the study system understandable from the documentation itself.

- `README.md` — what the repository is and how to use it.

### 2. Source-of-truth resource index

All study materials should eventually be catalogued by:

- subject;
- source;
- status;
- whether official / secondary / personal note / practice;
- review status;
- relevance to Bihar Judiciary.

### 3. Official-source preference

Where possible, the repository should prefer:

- BPSC sources for examination administration;
- India Code / official legislative sources for bare acts;
- Supreme Court and eCourts materials for official judicial-system references;
- PYQs and bare acts before coaching summaries.

### 4. Current criminal-law currency

The criminal-law section must prioritize the post-2024 laws:

- Bharatiya Nyaya Sanhita, 2023;
- Bharatiya Nagarik Suraksha Sanhita, 2023;
- Bharatiya Sakshya Adhiniyam, 2023.

IPC / CrPC / Evidence Act references may be used for comparison, but the active preparation map should not be IPC-only or CrPC-only.

### 5. Output-based learning

Each topic should produce at least one output:

- teacher notes;
- student answers;
- corrections;
- MCQ drill;
- mains skeleton;
- error-log entry;
- revision checklist.

## Practical Implementation Rule

- Sunday revision queue where required.
